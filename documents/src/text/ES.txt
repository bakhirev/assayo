§ report.logFile.title: Cómo descargar los datos desde git?
§ report.logFile.online.title: Para la visualización en línea
§ report.logFile.online.description: En el directorio raíz de su proyecto ejecutar:
§ 36:
git --no-pager log --raw --numstat --oneline --all --reverse --date=iso-strict --pretty=format:"%ad>%aN>%aE>%s" > log.txt
§ report.logFile.offline.title: Para ver sin conexión
§ 38:
git --no-pager log --raw --numstat --oneline --all --reverse --date=iso-strict --pretty=format:"%ad>%aN>%aE>%s" | sed -e 's/\\/\\\\/g' | sed -e 's/`/"/g' | sed -e 's/\$/S/g' | sed -e '1s/^/R(f\`/' | sed -e '$s/$/\`\);/' > log.txt
§ report.logFile.offline.description: Git creará un archivo `log.txt`. contiene los datos para construir el informe. La diferencia entre los formatos en línea y fuera de línea está en la existencia de una envoltura para las cadenas. El formato fuera de línea se cargará como un archivo`js` si usted simplemente lo abrió. `/build/index.html`

§ 44: Como mirar el reporte de un grupo de microservicios?
§ 45:
Generar para cada microservicio el archivo `log.txt` (`log-1.txt`, `log-2.txt`, `log-3.txt` etc.). You can do this manually, or use the [Assayo Crawler](https://github.com/bakhirev/assayo-crawler) module for automatic log collection.
Ver “Cómo ver el informe en línea?”. En el último paso, arrastrar todos los archivos al navegador.
Ver “Cómo ver informe fuera de línea?”. En el segundo paso, arrastrar los archivos de todos los microservicios al navegador (`log-1.txt`, `log-2.txt`, `log-3.txt` etc.) la carpeta informe (`/build`).
§ 46: THE BEST PRACTICES IN THE PROJECT
§ 47: ¿Cómo firmar los commits?
§ 48: Siga la práctica [Conventional Commits 1.0.0](https://www.conventionalcommits.org/en/v1.0.0/). Por ejemplo:
§ 49:
JIRA-1234 feat(profile): Added avatar for user 
§ 50:
El número de la tarea en el gestor de tareas es `(JIRA-1234)`
tipo de trabajo `(feat, fix, style, refactor, test, doc etc.)`
ficha `(profile - La sección del sitio, la página o la nueva funcionalidad, en una palabra)`
¿qué problema resolvieron? `(Added avatar for user)`
§ 51: How to add checking for commit message?
§ 52:  Use file `commit-msg`
§ 53: 1. Create file `commit-msg` in folder `.git/hooks/`
§ 54: 2. Add this text in file:
§ 55:
#!/usr/bin/env bash
if ! grep -iqE "(JIRA-[0-9]{1,5})(\s)(feat|fix|docs|style|refactor|test|chore)((\([a-z0-9_-]{1,}\)){0,})(:\s)([a-z]{1,})" "$1"; then
   echo "Need commit message like: JIRA-12 fix(profile): some text. Read Semantic Commit Messages" >&2
   exit 1
fi
§ 56:  Use package [pre-commit](https://www.npmjs.com/package/pre-commit)
§ 57: 1. Add in file `package.json` property `commit-msg`:
§ 58:
  ...
  "commit-msg": {
    "regex": "(JIRA-[0-9]{1,5})(\\s)(feat|fix|docs|style|refactor|test|chore)((\\([a-z0-9_-]{1,}\\)){0,})(:\\s)([a-z]{1,})",
    "error-message": "Need commit message like: JIRA-12 fix(profile): some text Read Semantic Commit Messages"
  },
  ...
§ 59: 2. Run command `npm install pre-commit`
§ 60: ABOUT THIS APP
§ 61: ¿Cómo puedo personalizar la interfaz de usuario?
§ 62: Puedes crear tu propio tema para la interfaz. Puedes cambiar:
§ 63:

**El título**. Puedes especificarlo en el parámetro de la URL ```title```. Por ejemplo: ```?title=You Company```

**El tema visual**. Para esto, debes preparar un archivo CSS con los nuevos estilos y especificar su dirección en el parámetro de URL ```theme```. Por ejemplo: ```?theme=//company.com/some.css```. Puedes usar los nombres de clases como selectores. La mayoría de ellos no cambia con el lanzamiento de nuevas versiones.

**La lengua**. Puedes especificarla en el parámetro de la URL ```lang```. Por ejemplo: ```?lang=es```
